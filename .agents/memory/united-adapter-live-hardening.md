---
name: United portal adapter — live-submit hardening & create-POST instrumentation
description: How United's multi-step ASP.NET/Salesforce create differs from SIT, and the redaction rule for capturing its create contract
---

United (partner.unitededucation.com/Manage/newapplication) is a Salesforce-backed **ASP.NET MVC + KT Stepper multi-step wizard**, NOT a single webhook like SIT — so it cannot be collapsed to one HTTP call. The adapter drives the UI and, on the first LIVE run, instruments `page.on("requestfinished")` to capture the real create contract for a future Phase-2 HTTP replay.

**Redaction rule (critical):** capturing the create POST body must reveal the CONTRACT (every field NAME + structural Salesforce ids: contactid/programid/appid + the cascade select ids) but NEVER applicant PII values. Rule = **default-deny on values**: log all keys, redact every value except a narrow structural allowlist (`*_id`/`id$`, `select{university,program,degree,lang,campus}`, regtype/country/destination/degree/program/lang/campus, `university*`), and always redact keys matching a PII regex (name|passport|kimlik|phone|mail|address|birth|dob|dateinput|school|father|mother|national|tckn). Handle form-urlencoded (the real create) AND JSON; unknown format → suppress entirely. Anti-forgery `__RequestVerificationToken` always redacted. Also strip URL query strings before logging (`url.split("?")[0]`) so `/Account/searchapp?word=<email>` doesn't leak. This mirrors SIT's "never log the request body (PII)" convention while still exposing the field names we need.

**Dry boundary:** `dryRun = doSubmit===false || PORTAL_DRYRUN==="1"`. Dry walks the KT-Stepper steps (term→degree→program cascade) and STOPS at the Program→Personal boundary — before personal-info fills and before the final submit. The Salesforce application record is created ONLY at the final submit, so no student is created in dry. This existing behavior is spec-locked ("Değiştirme") — do NOT add an aborting guard on step-transition POSTs; KT-Stepper "Continue" may POST to /Manage/newapplication per step and a guard would false-trip.

**Member gate:** submit() hard-skips any `profile.universityName` not in UNITED_ALLOWLIST (Biruni / Nişantaşı / Ankara Bilim — exactly 3, folded fuzzy) → `{skippedNotMember:true, routeTo:"direct"}`, same philosophy as SIT membership. Dedup via `GET /Account/searchapp?word=email|name` → parse `{totalSize,records}`, log count+Id only, seed `#contactid` when found.

**Program step reuses `matchProgram`** against the LIVE select2 options (panel nameMap/general + synonyms), captures `availablePrograms` + `resolution:"not_in_dropdown"` on miss. select2 is driven programmatically: set `el.value` + dispatch `change` + `jQuery(el).trigger("change")`, then wait for cascade AJAX.

**Program cascade matching (fixes `programMissing` false-positives):** each cascade `<select>` (university→program→degree→lang→campus) goes through `selById`, which WAITS for the AJAX-populated options (`waitForFunction`, 12s), LOGS the option texts, and matches flexibly via `looseMatchIndex` = `fold(strip "(...)")` then (a) normalized substring either-direction (reverse only len>4), (b) token-overlap ≥0.6. United labels carry parenthetical qualifiers the CRM name omits (e.g. `"Computer Engineering (Non-Thesis) (Turkish)"`) so plain `.includes()` never matched. Program step keeps `matchProgram` (panel-aware, thesis-aware) as the FIRST pass, then `looseMatchIndex` over live option TEXT as fallback — stripping parens can pick a Thesis/Non-Thesis sibling, accepted only because matchProgram runs first. **Critical selects** (`CRITICAL_SELECT_IDS` = selectuniversity/program/degree/lang) NEVER auto-pick the first option on a miss (would submit the wrong value) — only a LONE real option is forced. **Empty-target respect-preselection:** when a select is called with no target (selectlang/selectcampus), first honor an existing real `selectedIndex` (United derives degree/lang/campus from the program and auto-selects them) — this stops critical `selectlang` from stalling the Program→Personal step. Selection always fires `selectedIndex`+`change`+`jQuery.trigger("change")` then waits 1200ms for the next cascade.

**Member gate EN↔TR matching (`isUnitedMember`):** apply sends English names ("Istanbul Nisantasi University") but UNITED_ALLOWLIST is Turkish ("Nişantaşı Üniversitesi"); plain `fold`+substring never matched → wrongly routed to direct/SKIP (exclusive_region). Fix = `strip()` (fold + remove institution/city stopwords: university|universitesi|universite|univ|istanbul|the|of) then match by (a) name-contains-entry substring OR (b) token-subset (member iff ALL distinctive entry tokens appear in the name). CRITICAL: use ONLY `f.includes(e)` (name⊇entry), NOT the reverse `e.includes(f)` — the reverse false-positives non-member "Ankara University" (strips to "ankara") against entry "ankara bilim" (a real major non-member public uni). It's a strict 3-school gate so false-positives are worse than false-negatives; pure-translation variants like "Ankara Science University" (science≠bilim) intentionally won't match — add a curated alias only if seen in CRM data. Verify with the 10-case matrix (node --experimental-strip-types importing real fold).

**United portal redesigned → 6-step KT-Stepper wizard (2026-07-06):** the old single-page `#selectuniversity/#selectprogram/#selectdegree` cascade model was WRONG — it never advanced past screen 1, so every cascade select stayed empty → programMissing:true. Real flow (submit() drives it between goto /Manage/newapplication and the atPersonal/dryRun boundary): Step1 term radio `input[name="radio_buttons_2"]` (check+click+change, global `clickradio()`) → Continue; Step2 degree CARD radio (value=Bachelor/Master/PhD/Vocational School, click + global `clickradio1()`) → Continue; Step3a `#selectuniversity` select2 filter set to member uni (option values are ENGLISH accent-free e.g. "Nisantasi University") → wait grid AJAX; Step3b program is a CARD grid `div.single-table` (NOT a dropdown) — fuzzy-match title, click `span.plan-submit` (jQuery delegation, needs real .click()) → Continue. Registration Type `#regtype` MOVED to the Personal step (Step4) — it's no longer on the first page. Hidden Salesforce linkage `#contactid/#accountid/#appid` auto-fill as the wizard progresses — never seed them. Program-card match SAFETY (critical for live submits): accept only exact/near-exact normalized title containment (score 1) OR multi-token target with overlap ≥0.67 (single shared token must NOT pick a sibling like "Civil Engineering"); THEN hard-gate on the "Selected Majors (N)" counter incrementing after the click (fail-closed = programMissing stays true; a false-negative is safe, a false-positive submits the wrong program). Counter check is English-regex-coupled; if UI localizes, add a selected-CSS/disabled-button fallback signal.

**`__name is not defined` in page.evaluate:** esbuild/tsx wraps NAMED inner functions passed to page.evaluate with `__name(fn,"…")`, but `__name` doesn't exist in the browser context → any evaluate containing a named inner arrow (e.g. `const nrm=(s)=>...`) throws (Step-1 evaluate had no named inner fn so it worked). Fix = ensureNameShim() that does `page.evaluate(()=>{ globalThis.__name ||= (f)=>f; })`, called right after goto AND at the end of clickContinue (KT-Stepper swaps steps client-side w/o reload, but re-assert per step in case a remount clears globals).

**Step-3 program grid: wait for cards + preserve Thesis tokens.** Two headless-only bugs surfaced live: (1) fixed `waitForTimeout(2500)` read the grid before the ~150 cards rendered (AJAX, slower headless) → 0 cards, matched:false. Fix = `waitForFunction` (15s) gating on `div.single-table` present AND at least one card containing the university's distinctive core token (uniCore = longest token of folded universityName minus university/istanbul/etc.), then 1s settle; `.catch` fail-safe. (2) the program-match `nrm` was deleting parenthetical CONTENT (`replace(/\([^)]*\)/g," ")`), collapsing "(Thesis)" and "(Non-Thesis)" to the same string → wrong variant. Fix = drop only punctuation→space (keep tokens non/thesis/turkish) so the correct variant out-scores its sibling via token overlap. Card title shape: "<Uni> <Program> (<Lang>) (<Thesis|Non-Thesis>) <Degree>".

**Step-3 grid loaded by page's own filterData(), not the change event.** Live proof: the United program grid renders via a global `filterData()` (reads #selectuniversity/#selectdegree/#selectlang/#selectcampus). Setting the value + `jQuery(u).trigger('change')` works in a real browser but headless does NOT invoke filterData() off that change → 0 cards. Fix = after dispatching change, call the page's own loaders directly, each guarded by typeof + try/catch: `filterData()` and `updateProgramsAndCampuses()`. Pairs with the card-render waitForFunction that follows.

**Step-3 program cards render in the INACTIVE step container (hidden).** After filterData() loads the grid, the wizard has already advanced (Continue flow → Step 4 Personal, atPersonal=1), so the div.single-table cards render inside the now-inactive Step-3 container and are offscreen/hidden. The progPick `if (!c.offsetParent) continue;` guard skipped every hidden card → matched:false with 150 cards present. Fix = remove the visibility skip (scan hidden cards too); a programmatic .click() on a hidden span.plan-submit still fires and increments "Selected Majors" (the confirmation gate still guards). Also log `cardCount: cards.length` in both progPick return branches for diagnosis (surfaces in the step3 log).

**Step-3 program selection is alert9(checkbox.value), NOT span.plan-submit click.** The visible span.plan-submit has NO click handler — clicking it never increments "Selected Majors". Real mechanism: each card has a hidden input.plan-submit-checkbox whose value = the program's Salesforce id, with onclick="alert9('<id>')". Fix = find the card's input.plan-submit-checkbox; if already checked treat as selected, else call the global alert9(cb.value) (fallback cb.click()), guarded by try/catch. alert9 is global so it fires even when the card is hidden in the inactive step container. Live: alert9 took "Selected Majors (0)→(1)" and revealed Continue. The counter confirmation gate still validates.
