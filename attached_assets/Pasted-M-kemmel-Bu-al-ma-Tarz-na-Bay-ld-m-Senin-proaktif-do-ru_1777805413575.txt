Madde #1 — logAudit Fire-and-Forget
Önce: Doğrulama Adımları (sen uygulayacaksın)
Prompt'u çalıştırmadan önce şunları kontrol et — eğer varsayımlarımdan biri yanlışsa prompt'u uyarlayalım:

logAudit fonksiyonunun tanımı nerede? Tahminim: artifacts/api-server/src/lib/auth.ts içinde veya ayrı lib/audit.ts. Aç ve aşağıdakilere bak:

Fonksiyon imzası (parametreleri: userId, action, targetType, targetId, meta, ip gibi olabilir)
DB write'ı await db.insert(...) mu yoksa transaction içinde mi?
Caller'a herhangi bir return value (insert ID gibi) döndürüyor mu? Bu kritik: Eğer return value kullanılıyorsa fire-and-forget yapamayız.
Try-catch içinde mi, yoksa caller'a hata fırlatıyor mu?


Kaç callsite var? Replit search ile logAudit( ifadesini ara. Tahminim 50-150 arası. Eğer >200 ise prompt'u parça parça uygulamak gerekir.
await logAudit mı, yoksa void logAudit veya logAudit().catch(...) mı? Bazı callsite'lar zaten fire-and-forget olabilir — onları atla.
Audit log'un çıktısı nereye? Sadece DB tablosu mu, yoksa external service (CloudWatch/Sentry) de var mı? External varsa retry stratejisi düşünülmeli.

Replit Agent Prompt'u
GÖREV: artifacts/api-server/src/lib/auth.ts (veya logAudit fonksiyonu nerede tanımlıysa) içindeki logAudit fonksiyonunu fire-and-forget hâline getir. Caller'lar artık await etmek zorunda kalmasın.

YAPILACAKLAR:
1. logAudit fonksiyonunun mevcut imzasını koru. Async kalsın ama içinde yapılan DB insert'ini try-catch ile sarmalayıp hata olursa console.error("[audit] failed:", err, { userId, action, targetType, targetId }) ile logla, asla throw etme.

2. logAudit fonksiyonu Promise<void> döndürmeye devam etsin (signature breaking change yok). Ama implementasyonu içinde DB insert'inin sonucunu await EDERKEN bile dış caller'ı bekletmeyecek bir yardımcı çağrı modeli oluştur:

   - Yeni bir export: `fireAuditLog(...args)` — bu, `logAudit(...args).catch(err => console.error("[audit] failed:", err))` çağrısını yapar ve void döndürür (Promise döndürmez).

3. SADECE artifacts/api-server/src/routes/ altındaki tüm dosyalarda `await logAudit(` çağrılarını bul. Her birini şu kalıba çevir:

   ÖNCESİ:  await logAudit({ userId, action: "...", ... });
   SONRASI: fireAuditLog({ userId, action: "...", ... });

4. Eğer bir callsite şu üç durumdan birindeyse DEĞİŞTİRME ve atla:
   a) try-catch içinde özel hata yönetimi varsa (örn. audit fail edince işlem rollback ediliyorsa)
   b) Transaction içinde çağrılıyorsa (db.transaction(...) bloğu içinde)
   c) Return value'sı kullanılıyorsa (örn. const result = await logAudit(...))

5. Değişiklik yapılan her dosyanın import satırına gerekirse fireAuditLog ekle.

6. Hiçbir mevcut davranışı bozma. Audit log hâlâ DB'ye yazılmaya devam etmeli, sadece request thread'i bunu beklemeyecek.

DOĞRULAMA: Bitirince
- pnpm run typecheck çalıştır, hata olmamalı.
- Değiştirilen callsite sayısını ve atlanan callsite'ları (yukarıdaki 4. madde) raporla.
- logAudit içindeki hata fırlatma davranışının değiştirilip değiştirilmediğini açıkça söyle.

YAPMAMAN GEREKENLER:
- Mevcut audit log şemasını değiştirme.
- BullMQ/pg-boss gibi external queue ekleme — bu sprint'te kapsam dışı, sadece setImmediate/Promise.catch yeterli.
- logAudit fonksiyonunun signature'ını değiştirme.
- routes/ klasörü dışındaki yerlere dokunma (lib/ içindeki internal callsite'lar bu sprint dışı).
Beklediğim Riskler

Transaction içindeki callsite'lar: Eğer db.transaction(async tx => { ... await logAudit(...) ... }) varsa fire-and-forget yapmak audit log'un transaction commit'ten önce başlatılmasına yol açabilir. Bunlar atlanmalı (prompt'ta belirttim).
Test ortamı: Eğer e2e test'lerinde "audit log oluştu mu?" diye kontrol eden assertion varsa, fire-and-forget timing yarışına yol açar — Agent'ın bunu raporlaması gerek.


Madde #8 — Course Finder Hardening (Paralel)
Önce: Doğrulama Adımları

course-finder.ts'in search parametresinin gerçekten ilike ile mi kullanıldığını doğrula. İlk gördüğüm satırlarda direkt görmedim, scroll edip ilike( çağrısını teyit et.
feeMin ve feeMax için validation var mı bak. Yoksa NaN injection olabilir.
limit parametresinin gerçekten 1000 max olduğunu satırda doğrula (ben okudum: Math.min(1000, ...)).

Replit Agent Prompt'u
GÖREV: artifacts/api-server/src/routes/course-finder.ts dosyasında üç küçük güvenlik/dayanıklılık düzeltmesi yap.

YAPILACAKLAR:

1. limit cap'ini 1000'den 100'e indir.
   ÖNCESİ: const limitNum = Math.min(1000, Math.max(1, parseInt(limit, 10)));
   SONRASI: const limitNum = Math.min(100, Math.max(1, parseInt(limit, 10)));

2. search parametresi için ilike wildcard escape ekle. ilike çağrısının yapıldığı satırı bul (search query'sinde kullanılan). Şu yardımcı fonksiyonu dosyanın üstüne ekle:

   function escapeLikePattern(s: string): string {
     return s.replace(/\\/g, '\\\\').replace(/%/g, '\\%').replace(/_/g, '\\_');
   }

   Sonra search kullanılan ilike çağrısında pattern oluşturulurken escapeLikePattern uygula:
   ÖNCESİ:  ilike(programsTable.name, `%${search}%`)
   SONRASI: ilike(programsTable.name, `%${escapeLikePattern(search)}%`)

   search hangi sütunlarda kullanılıyorsa hepsinde aynı escape uygulansın.

3. feeMin ve feeMax için NaN guard ekle. Eğer parseFloat sonrası NaN ise koşulu eklemeden geç:
   ÖNCESİ:  if (feeMin) conditions.push(...)
   SONRASI: if (feeMin) {
              const n = parseFloat(feeMin);
              if (!isNaN(n) && isFinite(n) && n >= 0) conditions.push(...) /* n kullan */
            }
   feeMax için aynısı.

DOĞRULAMA:
- pnpm run typecheck — hata olmamalı.
- Manuel test: GET /api/course-finder?search=%25test%25 isteği atıldığında % artık literal olarak aranmalı, wildcard olarak değil.
- GET /api/course-finder?limit=500 isteği 100 sonuç döndürmeli.
- GET /api/course-finder?feeMin=abc isteği fee filter'ı uygulamadan tüm sonuçları döndürmeli (500 hatası vermemeli).

YAPMAMAN GEREKENLER:
- Diğer query parametrelerine dokunma.
- Response format'ını değiştirme.
- Yeni endpoint ekleme.
Beklediğim Riskler

Mevcut frontend limit=200 veya benzeri bir değer gönderiyorsa kırılır. Frontend'de bu cap'e bağımlılık var mı kontrol et — büyük ihtimalle yok (UI default 24 zaten görmüştüm: limit = "24"), ama embed widget veya admin sayfası farklı kullanıyor olabilir.
Escape fonksiyonu ile mevcut Drizzle query üreticisi çakışabilir. Drizzle bazen kendi parametrik escape'ini yapıyor. console.log ile generated SQL'i bir kez doğrula.
