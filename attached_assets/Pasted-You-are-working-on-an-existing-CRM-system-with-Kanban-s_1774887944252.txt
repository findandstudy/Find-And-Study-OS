You are working on an existing CRM system with Kanban-style boards for Leads, Students, and Applications.

Your task is to implement a robust, end-to-end “Origin / Source Ownership” system across Lead, Student, and Application records, without breaking any existing functionality, permissions, filters, or board behavior.

IMPORTANT:
There is already some kind of “lead source” or “channel” information in the Leads UI (examples: website, referral, embed, agent). Do NOT overload or replace that field.
We must clearly separate these two concepts:

1) Acquisition Channel = where the lead came from
   Examples: website, referral, embed, manual import, campaign, landing page
2) Business Origin / Ownership Source = who brought the person into the system
   Allowed values:
   - direct
   - agent
   - sub_agent

The goal is that every Lead, Student, and Application card clearly shows whether it is:
- Direct
- Agent
- Sub-Agent

and, when applicable, also shows the related organization/person name:
- Direct
- Agent · ABC Agency
- Sub-Agent · Karachi Office

==================================================
PHASE 1 — INSPECT THE CURRENT CODEBASE FIRST
==================================================

Before changing anything, inspect and understand:

- database schema / ORM models / migrations
- Lead model and Lead board/card components
- Student model and Student board/card components
- Application model and Application board/card components
- conversion flow from Lead -> Student
- creation flow from Student -> Application
- manual create forms
- bulk import flow
- role/permission model for main company, admin, staff, agent, sub-agent
- existing filtering/search logic
- existing audit log model if available

Then implement the feature in the cleanest possible way using the existing architecture and UI style.

==================================================
PHASE 2 — DATA MODEL CHANGES
==================================================

Add a new normalized origin system to Leads, Students, and Applications.

For each entity, add fields equivalent to the following (adapt to current schema naming conventions if needed):

- originType: enum('direct', 'agent', 'sub_agent')
- originEntityType: enum('company', 'agent', 'sub_agent') nullable if not needed
- originEntityId: nullable string/uuid/number
- originDisplayName: nullable string
- originLocked: boolean default true once inherited through conversion
- originLeadId: nullable on Student and Application if helpful for lineage
- originStudentId: nullable on Application if helpful for lineage

Do NOT remove or repurpose the existing lead source/channel field.

If the project already has fields like createdBy, agentId, subAgentId, tenantId, ownerId, panelOwnerId, or assignment relations, reuse them intelligently instead of duplicating logic.

==================================================
PHASE 3 — BUSINESS RULES
==================================================

Implement the following business rules consistently:

A) LEAD CREATION
1. If a lead is created by main company / internal staff / admin / super admin for the main panel:
   originType = 'direct'
   originDisplayName = 'Find And Study'

2. If a lead is created under an agent account:
   originType = 'agent'
   originEntityType = 'agent'
   originEntityId = that agent’s id
   originDisplayName = that agent’s company/brand name

3. If a lead is created under a sub-agent account:
   originType = 'sub_agent'
   originEntityType = 'sub_agent'
   originEntityId = that sub-agent’s id
   originDisplayName = sub-agent name (or office/branch name)

4. If a lead is imported in bulk, preserve or infer origin using the importer’s role unless an explicit origin mapping is provided.

B) LEAD -> STUDENT CONVERSION
When a Student is created by converting a Lead:
- copy the Lead’s entire origin metadata to Student
- lock it by default so it remains traceable
- store reference to source lead if useful

C) STUDENT CREATION WITHOUT LEAD
If a Student is created manually:
- by main company/internal staff => direct
- by agent => agent
- by sub-agent => sub_agent

D) STUDENT -> APPLICATION
When an Application is created from a Student:
- inherit the Student’s origin metadata exactly
- do not recalculate it from scratch
- store reference to student if useful

E) APPLICATION CREATED DIRECTLY
If your system allows direct manual Application creation:
- if tied to an existing student, inherit from student
- if created standalone, infer from current user role and ownership context

F) EDIT RULES
- origin should be read-only in normal workflows
- only super admin (and optionally admin) can manually override origin
- every manual override must be logged in audit trail with old value, new value, user, timestamp

PHASE 6 — BACKFILL / MIGRATION FOR EXISTING DATA
==================================================

Create a safe migration / backfill strategy for existing records.

For existing Leads:
- infer origin from creator role and existing ownership relations
- if owned by main company/internal accounts => direct
- if associated with an agent => agent
- if associated with a sub-agent => sub_agent

For existing Students:
- if linked to a Lead, inherit from linked Lead
- otherwise infer from creator/owner role and hierarchy

For existing Applications:
- if linked to a Student, inherit from Student
- otherwise infer from linked Lead/creator/owner context

Backfill rules should be deterministic and safe.
Avoid setting null origin unless absolutely impossible.
If impossible, set a conservative default and log/report exceptions.
