CORE DEFINITIONS

Define these clearly in code and UI:

1. Session Start
- when authenticated user opens the app and becomes active in a valid session

2. Session End
- explicit logout
- session timeout
- browser/tab close if detectable
- long inactivity timeout
- fallback close logic if abrupt disconnect happens

3. Active Time
A user should count as active only when they are actually interacting with the app.
Use signals such as:
- mouse movement
- clicks
- scrolling
- keyboard interaction
- route/page interaction
- window/tab focus
- page visibility
Do not overcount passive open-tab time as active time.

4. Idle Time
- if there is no interaction for a configurable threshold (for example 60 seconds or 120 seconds), mark user as idle
- when the user interacts again, mark them active again

5. Page/Module Time
Track:
- which route/page/module the user opened
- how long they stayed there
- how much of that time was active vs idle

6. Presence Status
Support internal presence states such as:
- active
- idle
- offline

ACTIVE TIME LOGIC
Implement a robust active-time model, not just login-to-logout duration.

Suggested approach:
- session starts
- app tracks interaction timestamps
- if no interaction after idle threshold, user becomes idle
- only focused/visible time with recent interaction counts as active
- hidden tab or blurred window should not continue counting as active
- session duration and active duration must be stored separately
